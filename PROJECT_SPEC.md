# PROJECT_SPEC.md

# Brim Expense Intelligence Copilot

## 1. Project Summary

Brim Expense Intelligence Copilot is a React + Supabase hackathon project that turns anonymized SMB corporate card transactions into an AI-powered finance operations workspace.

The platform helps a finance manager:

1. Ask natural-language questions about company spend.
2. Scan transactions against a digitized expense policy.
3. Review AI-assisted pre-approval recommendations.
4. Generate expense reports from recent transactions.
5. Detect anomalies and possible fraud using explainable risk scoring.

The project should feel like a serious finance SaaS product, not a toy chatbot. The AI should make the finance manager smarter by combining transaction data, policy context, risk signals, and approval workflows.

---

## 2. Confirmed Product Decisions

### Stack

- Frontend: React + Vite + TypeScript
- Database: Supabase Postgres
- Auth: Supabase Auth
- Storage: Supabase Storage for receipt and policy file artifacts
- Secure server-side logic: Supabase Edge Functions
- UI: Tailwind CSS + shadcn/ui
- Charts: Recharts
- CSV parsing: PapaParse in the browser
- AI reasoning: Claude Sonnet via Supabase Edge Functions
- Embeddings: OpenAI embeddings via Supabase Edge Functions
- Vector search: Supabase pgvector
- Reports: CSV export first
- Testing: unit tests for deterministic policy and risk engines

### Important Architectural Choices

- Use MCP-style internal tools, not a full MCP server for the first version.
- Use deterministic application logic for policy enforcement; the current implementation favors Python FastAPI for authoritative policy checks.
- Use AI for reasoning, summarization, query planning, and approval/report explanations.
- Use OpenAI embeddings only for policy RAG retrieval.
- Do not use Gemini Grounded Search for now.
- Do not let the AI invent totals, employee names, policy outcomes, or risk scores.
- Do not execute raw SQL generated by the AI unless it is inside the backend Talk to Data workflow, passes a dedicated read-only AI validator plus deterministic safety checks, and runs in a read-only transaction.
- Keep API keys and service-role secrets out of the frontend.

Current implementation note:

- The live repo now uses Python FastAPI as the primary backend/business-logic boundary.
- FastAPI owns deterministic policy services, risk, approvals, reports, and AI orchestration.
- React may keep transitional import and helper logic, but policy, risk, AI orchestration, approvals, and report logic should call FastAPI.
- Supabase Edge Functions may remain for compatibility or specialized secure tasks, but they are not the primary business-logic boundary.

## 3. Required Capabilities

The product must implement all four core capabilities.

### 3.1 Talk to Your Data

A finance manager can ask questions in plain English and receive summaries, tables, and charts.

Example questions:

- What did Marketing spend this month by category?
- Compare Engineering and Sales travel spend.
- Which merchants did we spend the most on?
- Which employees have the most missing receipts?
- Show high-risk transactions from September.
- Which departments are trending over budget?

Expected behavior:

- Claude converts the user question into a structured query plan.
- The app validates the plan.
- The app runs a safe query against Supabase.
- The answer is generated only from returned rows and summary statistics.
- Follow-up questions preserve prior context when appropriate.

The AI may suggest a visualization, but the backend/frontend decides how to render it.

---

### 3.2 Policy Compliance Engine

The app digitizes Brim’s expense policy into structured rules and scans transactions automatically.

The compliance engine must:

- Apply deterministic rules.
- Save policy checks.
- Save violations.
- Rank violations by severity.
- Surface repeat offenders.
- Identify transactions that need more context.
- Explain each violation using policy clauses retrieved from RAG where useful.

Claude may explain policy outcomes, but Claude must not be the policy enforcer.

---

### 3.3 AI Pre-Approval Workflow

When a transaction needs approval, the app creates an approval item with everything a manager or finance user needs.

Approval detail should include:

- Transaction details
- Employee
- Department
- Manager or approver
- Spend history
- Department budget status
- Policy flags
- Risk score and risk signals
- Relevant policy clauses
- AI-generated recommendation: approve or deny
- Readiness/context gaps and citations

The approver can:

- Approve
- Deny

Missing information is captured in the approval packet's readiness/context fields rather than as a decision outcome.

All decisions must be saved with an audit trail.

---

### 3.4 Automated Expense Report Generation

The app groups employee transactions into expense reports for a date range.

Reports should include:

- Employee and department
- Period start and end
- Line items
- Category grouping
- Total amount
- Missing receipts
- Missing pre-approvals
- Policy flags
- Risk flags
- AI-generated summary
- CSV export
- Workflow metrics, citations, visuals, and approval/readiness context

The report should be ready for a manager or CFO-style review.

## 4. Optional Capability to Implement Deeply

### Explainable Anomaly and Fraud Detection

The optional feature to prioritize is anomaly/fraud detection.

The first implementation should be deterministic and explainable. It should be structured so a real ML model can be added later.

Risk signals:

- Duplicate charges
- Split transactions
- Near-threshold behavior
- Round-number patterns
- Unusual merchant activity
- Merchant novelty for employee
- Merchant novelty for company
- Foreign transactions unusual for employee/department
- Employee/category peer deviation
- Department/category peer deviation
- Missing receipt
- Missing pre-approval
- Policy violation severity

Output should always explain why a transaction was flagged.

Example risk output:

```json
{
  "transaction_id": "txn_123",
  "risk_score": 87,
  "risk_level": "high",
  "signals": [
    {
      "type": "split_transaction",
      "severity": "high",
      "message": "Same employee made multiple same-day charges at the same merchant. Combined total exceeds the approval threshold."
    },
    {
      "type": "missing_preapproval",
      "severity": "high",
      "message": "Expense is over the policy threshold and has no approved pre-authorization."
    }
  ]
}
```

## 6. Synthetic Business Layer

### Decision

Use option C:

1. Seed employees, departments, managers, and budgets once.
2. Assign transactions deterministically during import.
3. Generate synthetic receipt and pre-approval statuses during import.

### Why

The required hackathon features need employees, departments, managers, budgets, receipts, and approvals. Since the raw CSV is transaction-only, these demo entities must be generated.

The app must clearly label these fields as synthetic/demo data. Do not pretend they came from the original CSV.

### Synthetic Company Structure

Seed approximately 50 employees across departments:

- Marketing
- Sales
- Engineering
- Operations
- Finance
- HR
- Customer Success
- Executive

Each department should have:

- Monthly budget
- Quarterly budget
- Department manager

Each employee should have:

- Name
- Email
- Department
- Manager
- Role
- Synthetic flag

### Stable Assignment

Transactions should be assigned deterministically so re-importing the same CSV produces stable results.

Example strategy:

```text
employeeIndex = stableHash(merchant + transactionDate + amount + sourceRowNumber) % employeeCount
```

This allows repeat offenders, employee history, and department trends to remain stable across demos.

### Hero Demo Scenario

Create one memorable demo scenario:

```text
Employee: Sarah Chen
Department: Marketing
Use case: conference/travel expense report
```

Sarah Chen should have enough transactions to demonstrate:

- Expense report generation
- Department budget context
- Missing receipt
- Transaction requiring pre-approval
- AI approval recommendation
- Policy explanation

## 7. Expense Policy Rules

Digitize these policy rules first.

| Rule Code | Rule | Severity |
|---|---|---|
| PREAPPROVAL_OVER_50 | Expenses over $50 require manager pre-authorization. | High |
| RECEIPT_REQUIRED | Receipts are required before reimbursement. | Medium/High |
| RECEIPT_CURRENT_MONTH | Receipts should be submitted within the current month where possible. | Low/Medium |
| FALSIFICATION_PROHIBITED | Falsified expense reports are prohibited. | Critical |
| ENTERTAINMENT_CONTEXT_REQUIRED | Customer entertainment requires guest names and business purpose. | Medium/High |
| ALCOHOL_RESTRICTED | Alcohol is not permitted unless dining with a customer. | High |
| TIPS_SERVICE_15 | Tips for services/porterage may be expensed up to 15%. | Medium |
| MEAL_TIPS_20 | Meal tips are not reimbursed above 20%. | Medium |
| PARKING_ALLOWED | Reasonable parking may be reimbursed. | Low |
| TOLLS_ALLOWED | Tolls may be reimbursed. | Low |
| TICKETS_NOT_REIMBURSABLE | Traffic or parking tickets are not reimbursed. | High |
| CAR_RENTAL_RECEIPTS_REQUIRED | Car rental, parking, and gasoline receipts are required. | Medium/High |
| CARD_NAMED_INDIVIDUAL_ONLY | Corporate cards may only be used by the named individual. | High |
| PERSONAL_CARD_USE_PROHIBITED | Personal expenses on corporate cards are prohibited. | Critical |
| CONSISTENT_ABUSE_REVIEW | Consistent abuse can restrict or revoke cards. | High |

The deterministic policy engine should enforce what can be enforced from available data and mark ambiguous cases as `needs_context`.

## 8. Transaction Normalization Rules

### Dates

- Parse `Posting date of transaction` and `Transaction Date` as dates.
- Expected source format is likely `M/D/YYYY` or `MM/DD/YYYY`.
- Store as ISO date in Supabase.

### Amounts

- `Transaction Amount` is the source amount.
- `Debit` means expense.
- `Credit` means refund/reversal.
- If `Conversion Rate` is `0`, empty, null, or invalid, use the transaction amount as CAD amount.
- Otherwise, compute:

```text
amount_cad = transaction_amount * conversion_rate
```

### Merchant Normalization

Normalize merchant names by:

- Uppercasing
- Trimming whitespace
- Removing long numeric fragments where appropriate
- Preserving meaningful words

Examples:

```text
FEDEX268575826 -> FEDEX
CAT SCALE COMPANY -> CAT SCALE COMPANY
```

Do not rely heavily on `Merchant City`; sample data may contain phone-number-like values in that field.

## 9. Categorization Strategy

Use deterministic merchant/MCC/category rules first.

Suggested mappings:

| Merchant / MCC / Keyword | Category |
|---|---|
| FEDEX, UPS, DHL, MCC 4215 | Shipping / Courier |
| AIR CANADA, DELTA, UNITED, AMERICAN, WESTJET | Airfare |
| UBER, LYFT, TAXI | Ground Transportation |
| HOTEL, MARRIOTT, HILTON, HYATT, AIRBNB | Lodging |
| RESTAURANT, CAFE, BAR, FOOD | Meals / Entertainment |
| STAPLES, OFFICE | Office Supplies |
| AWS, GOOGLE, MICROSOFT, OPENAI, SLACK, NOTION, GITHUB | Software / SaaS |
| GAS, SHELL, ESSO, PETRO, CHEVRON | Fuel |
| PARKING, TOLL | Parking / Tolls |
| LCBO, SAQ, LIQUOR, BEER, WINE | Alcohol / Restricted |
| TICKET, FINE, VIOLATION | Non-Reimbursable Fine |
| CAT SCALE COMPANY | Transportation / Fleet / Operations |
| Unknown | Uncategorized |

Category confidence:

- 0.95 for exact merchant match
- 0.85 for MCC match
- 0.70 for description keyword match
- 0.40 for unknown or fallback category

## 10. Policy RAG

### Goal

Use RAG to retrieve relevant policy clauses so Claude can explain compliance decisions with policy evidence.

### Provider

- OpenAI embeddings
- Supabase pgvector

### Embedding Defaults

- `OPENAI_EMBEDDING_MODEL=text-embedding-3-small`
- `OPENAI_EMBEDDING_DIMENSIONS=1536`
- `POLICY_RAG_TOP_K=5`
- `POLICY_RAG_MATCH_THRESHOLD=0.72`

### RAG Flow

```text
Seeded policy text
  -> chunk policy clauses
  -> embed chunks with OpenAI
  -> store vectors in Supabase pgvector
  -> embed retrieval query
  -> match policy chunks
  -> return relevant clauses to Claude
```

### Use RAG For

- Policy explanation
- Approval recommendation context
- Report summary context
- User questions about policy
- Explaining why a transaction was flagged

### Do Not Use RAG For

- Transaction totals
- Budget calculations
- Amount threshold checks
- Duplicate detection
- Split transaction detection
- Risk scoring math
- Compliance enforcement

## 11. MCP-Style Tool Layer

Do not build a real MCP server in the first implementation. Build MCP-style internal tools with stable contracts.

Suggested tools:

```text
spend.query
spend.dashboardSummary
policy.checkTransaction
policy.checkBatch
policy.retrieveClauses
risk.scoreTransaction
risk.scoreBatch
approval.createRequest
approval.decide
report.generate
report.exportCsv
```

The tool layer should live in:

```text
src/lib/tools/
```

The Claude-facing Edge Functions should use these contracts conceptually, but the browser should never call server-only secrets directly.

## 12. AI Behavior

Claude should be used for:

- Natural-language query planning
- Explaining spend results
- Explaining policy outcomes
- Approval recommendations
- Expense report summaries
- Clarifying missing information

Claude must not:

- Invent data
- Invent totals
- Invent employees
- Invent departments
- Invent policy results
- Execute SQL
- Override deterministic compliance results
- Call secret APIs from the browser

When data is missing, Claude should say what is missing.

Example:

```text
I can flag this as needing context, but I cannot determine whether the meal included a customer because guest names and business purpose are missing.
```

## 18. Acceptance Criteria

The project is acceptable when:

- App runs locally.
- User can upload CSV with the specified columns.
- Transactions are normalized and saved.
- Synthetic employees/departments are assigned.
- Dashboard totals update.
- Policy scan creates policy checks and violations.
- Risk scan creates explainable risk scores.
- Talk-to-data answers at least five useful questions.
- Approval request can be created from a transaction.
- AI recommendation uses tool/database outputs.
- Expense report can be generated for an employee/date range.
- Report can export CSV.
- Policy RAG retrieves relevant policy clauses.
- AI does not hallucinate numbers or override deterministic results.
- Unit tests cover policy and risk engines.

## 19. Demo Flow

Use this as the primary hackathon demo:

1. Finance manager logs in as the demo finance user.
2. User imports transaction CSV.
3. App seeds or confirms synthetic employees/departments.
4. Dashboard shows total spend, spend by department, open violations, and high-risk transactions.
5. User asks: “What did Marketing spend this month by category?”
6. App returns a chart, table, and summary.
7. User asks: “How does that compare to Engineering?”
8. App answers using follow-up context.
9. User runs compliance scan.
10. App flags missing receipts, missing pre-approvals, restricted alcohol, fines/tickets, and context-needed meals/entertainment.
11. User creates an approval request.
12. AI recommends approve or deny using policy, risk, budget, readiness, citations, and history.
13. User generates Sarah Chen’s Marketing expense report.
14. Report shows line items, policy flags, risk flags, workflow metrics, citations, visuals, approval/readiness context, summary, and CSV export.

## 21. Guiding Principle

Working end-to-end beats overengineering.

The strongest demo is:

```text
CSV import -> spend dashboard -> policy scan -> risk detection -> approval recommendation -> expense report
```

Every AI output should be grounded in tool/database results.
