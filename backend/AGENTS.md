# backend/AGENTS.md

This folder owns the Python FastAPI backend and is now the primary server-side business-logic boundary.

## Responsibilities

- Expose typed JSON endpoints for frontend workflows.
- Keep server-only Supabase, Claude, and OpenAI keys on the backend.
- Own the authoritative policy engine, transaction enrichment, risk, RAG, AI orchestration, approvals, and reports logic.
- Represent MCP-style tools as Python service/tool functions, not a real MCP server.
- Build approvals as complete decision packets for human approve/deny decisions only; do not add a request-information approval outcome.
- Build reports with workflow metrics, citations, visuals, CSV export, and approval/readiness context.

## Current backend expectations

- `routers/` stay thin.
- `services/` hold business logic.
- `tools/` hold tool-style orchestration.
- `schemas/` define stable API contracts.
- The backend may use the Supabase service role; React may not.

## Policy-specific rules

- The Python policy engine is the compliance source of truth.
- Return grouped transaction-level findings by default.
- Keep individual violations persisted internally.
- Use evidence wording such as "required" or "unavailable in the dataset" unless records explicitly prove a document is missing.
- Do not let receipt evidence dominate top-level compliance status.
- Prefer `business_category` first, then `normalized_category` as a compatibility fallback.
- Synthetic evidence generation must be deterministic. Never use random assignment.
- Policy Rules endpoints may expose manager-authoring shells, but deterministic execution and durable rule JSON evaluation belong in backend services, not routers or React pages.
- Configurable policy rules use `services/rule_evaluator.py` as the deterministic JSON-condition foundation. Keep this evaluator small, typed, and free of external rule-engine libraries; AI extraction may draft rule JSON later, but scan enforcement must validate and execute deterministic rules only.

## Configurable policy-rule architecture

- Keep routers thin; `backend/app/services/rule_evaluator.py` owns rule validation and deterministic rule execution when configurable rules are added.
- Rule JSON must use explicit allowlists for fields and operators. Never use `eval`, dynamic imports, user-provided Python, or generated SQL.
- Extracted AI rules are drafts by default. They must be persisted with review metadata and cannot affect `/policy/scan` until activated.
- Rule extraction prompts must send only policy text, optional company context, and the allowed/available field list. Do not send transaction rows or bulk findings to Claude for extraction.
- `/policy/scan` should load active rules once per scan or batch, build enriched transaction context, and persist grouped transaction-level results plus nested violation rows.
- AI extraction and explanation calls belong in backend services only. Full transaction scans must not call Claude or OpenAI for each transaction.
- During the refactor, avoid keeping hardcoded and configurable rules as independent sources of truth. If a hardcoded rule remains, mark it as fallback or seeded-rule compatibility in code/tests.

## Security and safety rules

- Do not expose `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_DB_URL`, `ANTHROPIC_API_KEY`, or `OPENAI_API_KEY` to React.
- Do not execute SQL generated by Claude or any user prompt.
- Do not let AI outputs override deterministic backend logic, except that Talk to Data may execute validated read-only SQL when the backend runs the dedicated SQL guard workflow and deterministic safety checks first.

## Migration rules

- Do not break or silently replace the existing React CSV import flow unless explicitly requested.
- When moving a frontend workflow into the backend, preserve the API contract or coordinate the contract update in the same change.
